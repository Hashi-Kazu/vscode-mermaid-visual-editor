## 1. 目的・スコープ

本文書は、Visual Studio Code向けMermaidエディタ拡張機能（`vscode-mermaid-visual-editor`）の要求仕様を定義する。

本拡張機能は、Markdownファイル（`.md`）内のMermaidコードブロックを視覚的に編集し、ソースコードと双方向に同期する手段を提供する。

### 4.1 US-C06 SVG / PNG エクスポート（実装済 v2.0）

**ユーザーストーリー**  
ユーザーは、ビューアでレンダリングした図を画像ファイルとして書き出したい。なぜなら、ドキュメントや資料に図を貼り付けたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-C06-01 | フローチャートパネルから現在のレンダリング結果をSVG / PNGでエクスポートできる | 必須 | ツールバー「⬇ エクスポート」または右クリックメニューから「SVGとして保存」「PNGとして保存」を選択できること |
| R-C06-02 | エクスポート時はVS Codeの保存ダイアログを表示し、保存先・ファイル名を指定できる | 必須 | デフォルトファイル名は元ファイル名の拡張子を置換したもの（例: `project.svg` / `project.png`）とすること |
| R-C06-03 | SVGはMermaid.jsが生成したSVG文字列を、PNGはCanvas経由でラスタライズしたバイナリを書き出す | 必須 | 保存後に書き出したパスを通知メッセージで表示すること |
| R-C06-04 | エクスポートには編集UI由来の要素・装飾を含めない | 必須 | 当たり判定用の透明パス（`fc-edge-hit`）、選択・ホバー等の一時装飾（`fc-selected` 等）、操作用に付与したインライン `cursor`/`pointer-events` を除去したクリーンなSVGを出力すること |

### 4.2 US-C09 複数人同時編集での変更消失防止（実装済 v2.3.0）

**ユーザーストーリー**  
ユーザーは、共有ドライブやGit経由で同じ `.md` を複数人が開いて編集する状況でも、自分のビューア操作が他者の変更を黙って上書きして消してしまわないことを期待する。なぜなら、リアルタイム共同編集はなくとも、知らないうちに他者の作業が失われるのは致命的だから。

本拡張のGantt／Flowchartビューアはいずれも編集操作のたびにドキュメント全文を再シリアライズして置換する（per-operationマージは行わない）。そのため、書き込みの直前に「ビューアの現在表示がどのドキュメント内容からパースされたか」を表す base スナップショットとライブ／ディスク内容を比較し、並行変更を検知する楽観的同時実行制御を適用する。判定ロジックは VS Code API に依存しない純粋関数として `src/conflictDetection.ts` に分離し、ユニットテスト（`test/conflictDetection.test.ts`）で検証する。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-C09-01 | ビューアの表示状態が「どのドキュメント内容（テキスト）からパースされたか」を base スナップショットとして記録する | 必須 | `_sendUpdate`（ready／外部変更同期）でビューアを更新するたびに、その時点のドキュメントテキスト（改行は LF に正規化）が base として保持されること。base は書き込み成功後にも書き込んだ内容で更新されること |
| R-C09-02 | 全文置換による書き込みの直前に、ライブのドキュメント内容およびディスク実体を base と比較してコンフリクトを検知する | 必須 | base と一致する場合のみ従来どおり書き込み＋保存する。base と不一致（他者が並行変更）の場合は黙って上書きしないこと。改行コードのみの差異（CRLF/LF）はコンフリクトとして扱わないこと。自分の書き込みエコー（書き込もうとしている内容と現状が一致）はコンフリクトとして扱わないこと。共有ドライブで `TextDocument` がディスクより古い場合に備え、`vscode.workspace.fs.readFile` によるディスク読み取りも判定に併用すること |
| R-C09-03 | コンフリクト検知時はモーダル警告（`showWarningMessage`）でユーザーに解決方法を選択させる | 必須 | 「最新を読み込む（自分の編集は破棄）」選択時は他者の変更を保持してディスクから再同期し、自分の書き込みを中止すること。「自分の変更で上書き（他者の変更は破棄）」選択時のみ従来どおり全文置換すること。ダイアログを閉じる（未選択）場合は安全側（最新を読み込む）として扱うこと |
| R-C09-04 | コンフリクト解決で破棄される側の内容を、衝突ファイルとしてバックアップ退避する | 推奨 | 上書き選択時は他者の版を、最新読み込み選択時は自分の版を、`<ファイル名>.conflict-<mine\|remote>-<timestamp>.md` として同じディレクトリに保存すること。バックアップ失敗はコンフリクト解決をブロックしないこと（ベストエフォート） |
| R-C09-05 | ビューア操作中（`_isOperating = true`）に届いた外部変更も、操作完了後の書き込み前検知で必ずコンフリクトとして扱う | 必須 | 操作中は外部変更イベントを無視する（R-G07-04／R-F06-04）が、その間 base は更新されないため、操作完了後の全文置換前に base 比較に乗り、外部変更が検知・分岐されること。既存の `_applyQueue` 直列化が壊れないこと |

**補足説明**
- Gantt（`_doApplyGanttData`）・Flowchart（`_doWrite`）・テンプレート挿入（`_insertTemplate`）の全書き込み経路を単一の書き込みプリミティブ `_writeDocument` に集約し、その内部でコンフリクト検知・解決を行う
- base は `EditorPanel._baseText`（改行 LF 正規化済み）として保持する。`null` の間（初回同期前）はコンフリクト扱いせず初回書き込みを許可する

## 5. ガントチャート機能要件

### 5.1 用語定義（ガントチャート固有）

| 用語 | 定義 |
|------|------|
| ガントパネル | ガントチャートをインタラクティブに操作するVS Code WebViewパネル |
| タスクバー | ガントチャート上で1タスクを示す横棒 |
| セクション | 複数タスクをグループ化する論理単位（`section` キーワード） |
| タスク | ガントチャート内の1行（ラベル・状態・ID・開始日・期間から構成） |

### 5.2 US-G01 ガントチャート表示

**ユーザーストーリー**  
ユーザーは、Mermaidのgantコードを開いたとき、インタラクティブなガントチャートを横並びで確認したい。なぜなら、日程の全体像を視覚的に把握しながら編集したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-G01-01 | Markdown内のgantブロックに対してガントパネルを開けること | 必須 | コマンド `mermaid.openGantt` またはエディタタイトルバーボタンからパネルが開くこと |
| R-G01-02 | ガントパネルはエディタの横（`ViewColumn.Beside`）に表示される | 必須 | 既に開いている場合は再利用すること |
| R-G01-03 | タスクを横棒（タスクバー）として時系列に描画する | 必須 | 開始日・期間に応じて正しい位置・幅で表示されること |
| R-G01-04 | セクション単位でタスクをグループ化して表示する | 必須 | セクション名が行ラベルとして表示されること |
| R-G01-05 | 日付軸（タイムライン）を上部に表示する | 必須 | 年月・週単位の目盛りが表示されること。表示範囲はタスク全体を包含すること |
| R-G01-06 | タスクの状態（`done` / `active` / `crit` / 未指定）を色で区別する | 必須 | 各状態に対して視覚的に異なるスタイルが適用されること |
| R-G01-07 | VS Codeのカラーテーマ（ダーク/ライト）に追従する | 推奨 | VS Code標準のCSS変数を使用していること |
| R-G01-08 | Mermaidコードの変更から 500ms 以内にパネルが更新される | 必須 | デバウンス処理を実装し、連続入力中は更新を遅延させること |
| R-G01-09 | ガントパネル上の操作完了時に `.md` ファイルを自動保存する | 必須 | タスク移動・リサイズ・名称変更・追加・削除のいずれの操作後も、VS Codeのダーティマーカーが残らないこと。セクション並び替え・Undo等の構造編集（structuralEdit）後も同様に保存されること |
| R-G01-10 | タスクバーのラベルが省略（…）表示されている場合、マウスオーバーでツールチップに全文を表示する | 必須 | バー幅に収まっているラベルにはツールチップを表示しないこと |
| R-G01-11 | ガントブロックが存在しないファイルをパネルで開いた場合、「ガントチャートを挿入」ボタンを表示する | 必須 | ボタンクリックで本日起点のデフォルトテンプレート（タイトル・2タスク）がファイルに挿入され、即座にガントパネルに反映されること。挿入と同時にファイルが自動保存されること |

### 5.3 US-G02 タスクバーのドラッグ移動（日程変更）

**ユーザーストーリー**  
ユーザーは、タスクバーを左右にドラッグして開始日を変更したい。なぜなら、コードを直接編集せず直感的にスケジュールを調整したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-G02-01 | タスクバーを左右にドラッグして開始日を変更できる | 必須 | ドラッグ中にバーがリアルタイムで移動すること |
| R-G02-02 | ドラッグ中は変更後の開始日をツールチップで表示する | 必須 | `dateFormat` に従った形式で日付が表示されること |
| R-G02-03 | ドラッグ完了時にMermaidコードの開始日が更新される | 必須 | タスクが `after <id>` 形式の相対指定の場合は絶対日付に変換して書き込むこと |
| R-G02-04 | スナップ機能：ドラッグ時に最小単位（1日）でスナップする | 必須 | 半日未満のドラッグは前日、半日以上は翌日にスナップすること |
| R-G02-05 | `Shift` キーを押しながらドラッグすると週単位でスナップする | 推奨 | 7日単位で吸い付く動作となること |

### 5.4 US-G03 タスクバーのリサイズ（期間変更）

**ユーザーストーリー**  
ユーザーは、タスクバーの右端をドラッグして期間（終了日）を変更したい。なぜなら、作業量の調整をビジュアルで行いたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-G03-01 | タスクバー右端のリサイズハンドルをドラッグして期間を変更できる | 必須 | リサイズハンドルはバー右端6pxの領域とし、カーソルが `ew-resize` に変わること |
| R-G03-02 | 最小期間は 1日 とする | 必須 | 1日未満に縮小しようとした場合は 1日 に固定すること |
| R-G03-03 | リサイズ中は変更後の終了日をツールチップで表示する | 必須 | `dateFormat` に従った形式で日付が表示されること |
| R-G03-04 | リサイズ完了時にMermaidコードの期間（`nd` / `nw` 等）が更新される | 必須 | 日数で計算し `Xd` 形式で書き込むこと |

### 5.5 US-G04 タスクのインライン編集・状態変更

**ユーザーストーリー**

| ID | 要件 | 優先度 | 受入条件 |

| R-G04-05 | タスクバーの状態インジケーター（バー左端のカラー部分）をクリックすると状態選択ピッカーが表示される | 必須 | タスク名のダブルクリック編集とは独立した操作として動作すること |

| R-G04-07 | 状態を選択するとMermaidコードの状態キーワードが更新され、タスクバーの色が即時変わる | 必須 | 未設定を選んだ場合はコードから状態キーワードが除去されること |

### 5.8 US-G07 Undo / 双方向同期の競合管理

**ユーザーストーリー**  
ユーザーは、ガントパネル上の操作を `Ctrl+Z` で取り消したい。また、テキストエディタとガントパネルを同時に編集しても壊れないことを期待する。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-G07-01 | Webview内に最大50操作のUndoスタックを保持する | 必須 | 51操作目以降は最古のエントリーが破棄されること |

| R-G07-03 | `Ctrl+Z` でMermaidコードをスナップショット時点の内容に復元する | 必須 | Undo後にテキストエディタの内容もスナップショット時点の状態になること |
| R-G07-04 | ガントパネルの操作中（`isOperating = true`）はMermaidコード変更イベントを無視する | 必須 | ドラッグ・リサイズ中にコード変更イベントが割り込んでもパネルが中断されないこと |
| R-G07-05 | 操作完了後、MermaidコードをソースオブトゥルースとしてパネルをReparse・再描画する | 必須 | 操作完了後、コードと同期した最新の状態がガントパネルに反映されること |
| R-G07-06 | テキストエディタ側で直接コードを変更した場合、500ms後にパネルが最新状態に更新される | 必須 | デバウンス後にパースしてパネルに反映されること |

## 6. フローチャート機能要件

### 6.1 用語定義（フローチャート固有）

| 用語 | 定義 |
|------|------|
| フローチャートパネル | フローチャートをインタラクティブに操作するVS Code WebViewパネル |
| ノード | フローチャート内の1要素（ラベル・IDから構成） |
| エッジ | 2ノードを結ぶ矢印または線（ラベルを持てる） |
| ポート | エッジの起点となるノードの端点UI（ホバー時に出現） |
| グラフ方向 | フローチャートのレイアウト方向（TD/LR/BT/RL） |
| オートレイアウト | Mermaid.jsが自動決定するノード配置 |

### 設計原則（SVGオーバーレイ方式）

フローチャートパネルはMermaid.jsが生成するSVGをそのままレンダリングし、その上にインタラクティブUI（クリックハンドラ・ポートハンドル・選択枠）をオーバーレイとして重ねる。ノードの物理的な位置はMermaid.jsのオートレイアウトが決定し、Mermaidコードにカスタム座標を保存しない。これにより`.md`ファイルが標準Mermaid記法を維持する。

### MD記述更新ポリシー（遅延分離）

Mermaidは「ノードをエッジ行へインライン記述する形式（`A[開始] --> B{条件}`）」と「ノードを単独行で宣言しエッジはID参照のみとする形式（`A[開始]` / `A --> B`）」の双方を標準としてサポートする。本拡張は後者（分離形式）を**正の更新ポリシー**とし、次の方針で書き込む。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-FP-01 | エッジの追加は常にID参照のみで行う | 必須 | `addEdge` は `A --> B` 形式で追記し、ノードの囲み記号をエッジ行へ書き込まないこと |
| R-FP-02 | ノードのラベル編集・形状変更は、対象ノードを単独宣言行へ集約してから反映する（遅延分離） | 必須 | 対象ノードのエッジ行インライン定義（囲み）を剥がしてID参照化し、ラベル／形状を保持する単独宣言行が**ちょうど1つ**になること。同一ノードが複数エッジにインライン定義されていても不整合が生じないこと |
| R-FP-03 | 編集対象でない既存行・サブグラフ・コメントは保持する | 必須 | 触れていないノードのインライン記述、`subgraph`〜`end`、`%%` コメント、行の並びを書き換えないこと（全面正規化は行わない） |

### 6.2 US-F01 フローチャート表示

**ユーザーストーリー**  
ユーザーは、Mermaidのflowchart/graphコードを開いたとき、インタラクティブなグラフを横並びで確認したい。なぜなら、フローの全体像を視覚的に把握しながら編集したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-F01-01 | Markdown内の`flowchart`/`graph`ブロックに対してフローチャートパネルを開けること | 必須 | コマンド `mermaid.openEditor`（自動判別）または `mermaid.openFlowchart`（明示指定）でパネルが開くこと |
| R-F01-02 | GanttチャートとフローチャートはひとつのEditorPanelを共有し、ツールバーの切り替えボタンで表示を切り替えられること | 必須 | 同時に2つのパネルが開かないこと。切り替え時にHTMLが再構築されること |
| R-F01-03 | Mermaid.jsが生成するSVGをWebview内にレンダリングする | 必須 | `flowchart TD/LR/BT/RL` および `graph TD/LR` 記法のいずれも描画できること |
| R-F01-04 | Mermaidコードの変更から500ms以内にパネルが更新される | 必須 | デバウンス処理を実装し、連続入力中は更新を遅延させること |
| R-F01-05 | パースエラー時はエラーメッセージをパネル内に表示する | 必須 | 最後の正常状態の描画を維持しエラー詳細を重ねて表示すること |
| R-F01-06 | flowchartブロックが存在しないファイルをパネルで開いた場合、「フローチャートを挿入」ボタンを表示する | 必須 | ボタンクリックでデフォルトテンプレート（4ノード・3エッジ）がファイルに挿入され、即座にパネルに反映されること。テンプレートは遅延分離ポリシー（R-FP-01〜02）に従い、ノードを単独宣言・エッジをID参照のみで記述すること。挿入と同時にファイルが自動保存されること |
| R-F01-07 | フローチャートパネル上の操作完了時に`.md`ファイルを自動保存する | 必須 | ノード・エッジいずれの操作後も、VS Codeのダーティマーカーが残らないこと |
| R-F01-08 | パネル起動後の初回レンダリング時、全ノードが画面に収まるよう自動フィット表示する | 必須 | 初回`update`メッセージ受信後50ms以内に`fitView()`が実行され、全ノードが可視領域内に収まること。2回目以降のレンダリング（Undo・テキスト編集による更新）では自動フィットしないこと |

### 6.3 US-F02 ノードのインライン編集

**ユーザーストーリー**  
ユーザーは、パネル上でノード名を直接変更したい。なぜなら、コードを見ずに図の内容を修正したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-F02-01 | ノードをダブルクリックしてラベルをインライン編集できる | 必須 | ダブルクリックでノード上にテキスト入力フィールドが表示されること。入力欄の文字サイズ・幅は現在のズーム倍率に緩やかに追従し（固定サイズで小さく見えず、かつ過大にもならないこと。平方根スケール・上限26px程度）、対象（ノード／ラベル）の中央に配置されること |
| R-F02-02 | Enterキーで確定、Escapeキーでキャンセルできる | 必須 | 確定後にMermaidコードのノードラベルが更新されること。Escape押下で変更を破棄し元のテキストに戻ること。更新は遅延分離ポリシー（R-FP-02）に従い、対象ノードを単独宣言行へ集約すること |
| R-F02-03 | 空文字列への変更は無効とする | 必須 | 空文字でEnterを押した場合は変更を適用しないこと |
| R-F02-04 | インライン編集中に入力欄以外の操作が起きたら編集を確定する | 必須 | 入力欄の外をクリック、背景パン、ホイールズーム、リセット（全体表示）など、入力欄と図形の表示位置がずれる操作の直前に編集内容を確定すること。パン用の `preventDefault` で blur が抑止される場合も確実に確定すること |

### 6.4 US-F03 ノードの追加・削除

**ユーザーストーリー**  
ユーザーは、パネル上でノードを追加・削除したい。なぜなら、コードを書かずにフローの構成要素を増減させたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-F03-01 | ツールバーの「+ ノード追加」ボタンまたは右クリック「ノードを追加」でノードを追加できる | 必須 | デフォルトシェイプは矩形、ラベルは「新しいノード」、IDは自動付番。操作後にMermaidコードが更新され、パネルに新ノードが描画されること |
| R-F03-02 | 追加直後にラベルのインライン編集が自動起動する | 必須 | R-F02-01 と同等の編集動作が即座に起動すること。再レンダリング完了後に編集フィールドが表示されること |
| R-F03-03 | 右クリック「ノードを削除」またはノードをクリック選択後に `Delete` キーでノードを削除できる | 必須 | ノードをクリックすると選択状態（強調表示）になること。削除後にMermaidコードからノード行および当該ノードに接続されているエッジ行が除去されること |
| R-F03-04 | コンテキストメニューはビューポートの端で折り返し表示する | 必須 | メニューが画面下端・右端に近い場合、表示位置を上または左にシフトして画面内に収まること |

### 6.5 US-F04 エッジの追加・削除・ラベル編集

**ユーザーストーリー**  
ユーザーは、ノード間の接続をパネル上で操作したい。なぜなら、矢印の接続関係を視覚的に確認しながら設定したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-F04-01 | ノードにマウスオーバーするとポート（端点ハンドル）が表示される | 必須 | ノード端付近にポートハンドルが出現し、カーソルが十字に変わること。ポートのドット中心が図形の各辺の中点に正確に揃い、ズーム倍率を変えてもズレないこと。再レンダリング・パン・ズーム時にポートが画面に取り残されない（確実に消える）こと |
| R-F04-02 | ポートをドラッグして別ノード上でドロップするとエッジを追加できる | 必須 | ドラッグ中は仮の矢印線が表示されること。ドロップ時にMermaidコードにエッジ行が追加されること。線種は R-F04-10 の既定エッジ設定に従うこと |
| R-F04-10 | 新規エッジの既定線種をツールバーで設定できる | 必須 | ツールバーの「既定エッジ」セレクタで実線矢印/点線矢印/太線矢印/矢印なし実線/矢印なし点線を選べ、ポートドラッグで追加するエッジがその線種で生成されること。この設定はMermaidコード（`.md`）には保存できないため**ビューア表示中のみ保持**する一時設定とし、`vscode.getState/setState` でパネルの再表示（hide/show・種別切替）をまたいで維持するが、パネルを閉じて開き直した場合は既定（実線矢印）に戻ること |
| R-F04-03 | エッジラベル部分またはエッジ線のダブルクリックでラベルをインライン編集できる | 必須 | 確定後にMermaidコードが `A -->|ラベル| B` 形式で更新されること。ラベル未設定のエッジ（`.edgeLabel` が潰れている場合）でも、エッジ線のダブルクリックでエッジ中央付近に編集フィールドが必ず表示されること |
| R-F04-04 | エッジラベルを空文字で確定するとラベルが除去される | 必須 | `A -->|ラベル| B` から `A --> B` に更新されること |
| R-F04-05 | エッジをクリック選択後に `Delete` キーまたは右クリック「エッジを削除」でエッジを削除できる | 必須 | エッジをクリックすると選択状態（強調表示）になること。MermaidコードからエッジR-F04行が除去されること |
| R-F04-09 | エッジの選択・編集の当たり判定を実線幅より広く取る | 必須 | エッジ線に沿った透明な太いヒット領域（約14px相当、ズームに追従）を重ね、細い実線上でなくてもクリック選択・ダブルクリック編集・右クリックができること。ヒット領域上の操作で背景パン・ノード追加が誤発火しないこと |
| R-F04-06 | エッジのラベル編集・スタイル変更・削除は対象エッジの始点／終点ノードを正しく特定して反映する | 必須 | Mermaid 11 系では `edgePath` の `LS-`/`LE-` クラスが固定値（`a1`/`b1`）となるため、これに依存せず `path` の `id`（`L_{始点}_{終点}_{連番}` 形式）から始点・終点ノードを解決すること。複数エッジが存在しても意図したエッジのみが更新されること |
| R-F04-07 | エッジ操作は始点→終点の向きまで一致する行のみを対象とする | 必須 | エッジ行の端点を実際にパースし、`from→to` の向きが一致するエッジのみを変更・削除すること。逆向き（`B-->A`）や両ノードを参照するだけの無関係な行を取り違えないこと。並行エッジは登場順のインデックスで一意に特定すること |
| R-F04-08 | エッジラベル要素とエッジの対応付けはDOM順インデックスに依存せず確実に行う | 必須 | Mermaid 11 はラベルの内側 `<span>` にも `edgeLabel` クラスを付与するため、`.edgeLabel` 数がエッジ数の2倍になりDOM順では崩れる。ラベルグループの `data-id`（`L_{始点}_{終点}_{連番}`）から対象エッジを解決し、ラベルのダブルクリック編集が正しいエッジを対象とし、インライン入力欄が当該ラベル位置に表示されること。ラベル未設定エッジはエッジ線のダブルクリックでエッジ中央に編集欄を表示すること |

### 6.7 US-F06 Undo / 双方向同期の競合管理

**ユーザーストーリー**  
ユーザーは、フローチャートパネル上の操作を `Ctrl+Z` で取り消したい。また、テキストエディタとパネルを同時に編集しても壊れないことを期待する。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-F06-01 | Webview内に最大50操作のUndoスタックを保持する | 必須 | 51操作目以降は最古のエントリーが破棄されること |
| R-F06-02 | 各操作（ノード追加・削除・ラベル編集・エッジ追加・削除・方向変更）の直前にスナップショットをUndoスタックに積む | 必須 | 1回の `Ctrl+Z` で1操作分だけ戻ること |
| R-F06-03 | `Ctrl+Z` でMermaidコードをスナップショット時点の内容に復元する | 必須 | Undo後にテキストエディタの内容もスナップショット時点の状態になること |
| R-F06-04 | フローチャートパネルの操作中はMermaidコード変更イベントを無視する | 必須 | 操作中にコード変更イベントが割り込んでもパネルが中断されないこと |
| R-F06-05 | テキストエディタ側で直接コードを変更した場合、500ms後にパネルが最新状態に更新される | 必須 | デバウンス後にパースしてパネルに反映されること |

### 6.8 US-F07 ビューのパン・ズーム

**ユーザーストーリー**  
ユーザーは、フローチャートパネル内でビューを移動・拡大縮小したい。なぜなら、大きなフローチャートでも全体を把握しながら作業したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-F07-01 | 背景（ノード・エッジ以外の領域）を左クリックドラッグでビューを水平・垂直に移動できる | 必須 | ノードのドラッグ・ポートのドラッグと競合しないこと |
| R-F07-02 | マウスホイールで拡大・縮小できる | 必須 | カーソル位置を中心にズームすること。`Ctrl` キーは不要 |
| R-F07-03 | フローチャートパネル内でのスクロールはズームとして動作する | 必須 | ホイール上: 拡大、ホイール下: 縮小。パネル外へのスクロール伝播は抑止すること |
| R-F07-04 | ツールバーの「リセット」ボタンで全ノード・全エッジが画面に収まる範囲で最大限に拡大し、中央に配置する | 必須 | ボタン名称は「リセット」とすること。実際に描画されたSVGのジオメトリ（画面上の実寸・位置）に基づき、全ノード・全エッジ（ラベル含む）が可視領域に収まる範囲で取り得る最大倍率に自動調整され、かつ図形全体の中心が可視領域の中央に正確にセンタリングされること。端の余白は最小限とし図形がビューを十分に埋めること（参照: R-F01-08 初回起動時の自動フィット） |
| R-F07-05 | ズームの最小倍率・最大倍率を設ける | 必須 | 最小 0.1 倍とすること。基本の最大倍率は 12.0 倍。ただし小さな図ではリセット時のフィット倍率が 12.0 倍を超えることがあり、その場合はリセット後もホイールでさらに拡大できるよう、ホイールズームの実効上限を「リセット時の表示倍率 × 4」へ引き上げること（リセット時の表示倍率自体は変えない）。最小倍率（縮小側）はホイールズーム・リセット共通とすること |
| R-F07-06 | 拡大してもノード・エッジ・ラベルの文字がぼやけずにくっきり描画される | 必須 | キャンバスを合成レイヤーへ昇格させてビットマップ拡大しないこと。ベクター SVG / foreignObject をズーム毎に再ラスタライズし、どの倍率でも文字が鮮明であること |

## 8. アーキテクチャ概要

### 8.1 共通アーキテクチャ

全ダイアグラム型ビューアは以下のパターンで実装する。新しいダイアグラム型を追加する際はこの構造に沿って対応ファイルを追加する。

```
┌──────────────────────────────────────────────────────────┐
│  VS Code Extension Host (Node.js)                         │
│                                                           │
│  extension.ts ─── コマンド登録・アクティベーション          │
│       │                                                   │
│  [Type]Panel.ts   ─── WebviewPanel 管理                   │
│  [Type]Parser.ts  ─── Mermaidコード → データモデル         │
│  [Type]Serializer.ts ─ データモデル → Mermaidコード        │
│                    │                                      │
│                postMessage / onDidReceiveMessage           │
└──────────────────────────────────────────────────────────┘
                        │
             (Sandboxed iframe)
┌──────────────────────────────────────────────────────────┐
│  Webview (Browser Context)                                │
│                                                           │
│  [type].js  ─── レイアウト計算・描画・操作制御              │
│  [type].css ─── スタイル                                  │
└──────────────────────────────────────────────────────────┘
```

### 8.2 ガントチャート メッセージプロトコル

| 方向 | タイプ | ペイロード | 用途 |
|------|--------|-----------|------|

### 8.3 フローチャート メッセージプロトコル

フローチャートは「SVGオーバーレイ方式」（§6.1 設計原則）を採用するため、拡張機能はパース済みデータではなく**生のMermaidコード（`rawCode`）**をWebviewに渡し、Webview側がMermaid.jsで描画する。各編集操作は文字列レベルでコードを書き換える。

| Webview → 拡張機能 | `ready` | `{}` | Webview初期化完了通知 |
| Webview → 拡張機能 | `initFlowchart` | `{}` | 空ファイルへデフォルトテンプレート挿入 |

| Webview → 拡張機能 | `addNode` | `{ x?, y? }` | ノード追加 |

## 9. データモデル

### 9.1 ガントチャート

```typescript
interface GanttData {
  title: string;
  dateFormat: string;        // 例: "YYYY-MM-DD"
  axisFormat?: string;       // 例: "%m/%d"
  sections: GanttSection[];
}

interface GanttSection {
  name: string;              // セクション名（セクションなしの場合は空文字）
  tasks: GanttTask[];
}

interface GanttTask {
  id: string;                // タスクID（省略時は自動付番）
  label: string;             // 表示ラベル
  status: 'done' | 'active' | 'crit' | 'milestone' | '';
  startDate: string;         // 絶対日付（dateFormat準拠）
  duration: number;          // 日数
  afterId?: string;          // 依存関係: `after <id>`（US-G11）
}
```

### 9.2 フローチャート

```typescript
type NodeShape = 'rect' | 'round' | 'diamond' | 'stadium' | 'circle';

type EdgeStyle =
  | 'solid-arrow'      // A --> B
  | 'dotted-arrow'     // A -.-> B
  | 'thick-arrow'      // A ==> B
  | 'solid-no-arrow'   // A --- B
  | 'dotted-no-arrow'; // A -.- B

interface FlowchartData {
  direction: 'TD' | 'LR' | 'BT' | 'RL';
  keyword: 'flowchart' | 'graph';
  nodes: FlowchartNode[];
  edges: FlowchartEdge[];
}

interface FlowchartNode {
  id: string;
  label: string;
  shape: NodeShape;          // US-F08
}

interface FlowchartEdge {
  id: string;                // `${from}::${to}::${idx}` 形式の内部識別子
  from: string;
  to: string;
  label?: string;
  style?: EdgeStyle;         // US-F09
}
```
