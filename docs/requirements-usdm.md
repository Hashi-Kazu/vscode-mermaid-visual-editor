## 前文ゾーン

### 目的・スコープ

本文書は `requirements.md`（MMD-REQ-001）をUSDM形式に変換した要求仕様書である。Visual Studio Code向けMermaidエディタ拡張機能（`vscode-mermaid-visual-editor`）の機能要件・非機能要件を定義する。

本拡張機能は、Markdownファイル（`.md`）内のMermaidコードブロックを視覚的に編集し、ソースコードと双方向に同期する手段を提供する。

**対象外**:
- Marketplace への公開・配布プロセス（GitHub Actions が担当）
- VS Code Extension API そのものの仕様
- Mermaid.js のレンダリング仕様詳細

## ＜共通機能＞

### R-C06 SVG / PNG エクスポート #US-C06 #実装済v2.0

> **理由：** ビューアでレンダリングした図を、ドキュメントや資料に貼り付けるため画像ファイルとして書き出す手段が必要だから。

> **説明：** フローチャートパネルのツールバー「⬇ エクスポート」または右クリックメニューからSVG/PNGを選択する。デフォルトファイル名は元ファイル名の拡張子を置換したもの（例: `project.svg` / `project.png`）。デフォルト出力パスの導出（拡張子置換）と書き込みデータの符号化方式判定（svg=utf-8 / png=base64）は VS Code API 非依存の純粋関数として `src/svgExport.ts`（`exportDefaultPath` / `exportEncoding`）に分離し、ユニットテスト（`test/svgExport.test.ts`）で検証する。DOM 依存のクリーンSVG生成（R-C06-04: `cloneNode`/`querySelectorAll`/`XMLSerializer`）は Webview 側 `media/flowchart.js` の `serializeCleanSvg` に残す。

###### ＜エクスポート操作＞

- ■■□ R-C06-01 フローチャートパネルから現在のレンダリング結果をSVG / PNGでエクスポートできる。ツールバー「⬇ エクスポート」または右クリックメニューから「SVGとして保存」「PNGとして保存」を選択できること
- ■■■ R-C06-02 エクスポート時はVS Codeの保存ダイアログを表示し、保存先・ファイル名を指定できる。デフォルトファイル名は元ファイル名の拡張子を置換したもの（例: `project.svg` / `project.png`）とすること（テスト済）
- ■■□ R-C06-03 SVGはMermaid.jsが生成したSVG文字列を、PNGはCanvas経由でラスタライズしたバイナリを書き出す。保存後に書き出したパスを通知メッセージで表示すること
- ■■□ R-C06-04 エクスポートには編集UI由来の要素・装飾を含めない。当たり判定用の透明パス（`fc-edge-hit`）、選択・ホバー等の一時装飾（`fc-selected` 等）、操作用に付与したインライン `cursor`/`pointer-events` を除去したクリーンなSVGを出力すること

### R-C09 複数人同時編集での変更消失防止 #US-C09 #実装済v2.3.0

> **理由：** 共有ドライブやGit経由で同じ `.md` を複数人が開いて編集する状況でも、ビューア操作が他者の変更を黙って上書きして消してしまわないことを保証するため。

> **説明：** 本拡張のGantt／Flowchartビューアはいずれも編集操作のたびにドキュメント全文を再シリアライズして置換する（per-operationマージは行わない）。そのため、書き込みの直前に「ビューアの現在表示がどのドキュメント内容からパースされたか」を表す base スナップショットとライブ／ディスク内容を比較し、並行変更を検知する楽観的同時実行制御を適用する。判定ロジックは VS Code API に依存しない純粋関数として `src/conflictDetection.ts` に分離し、ユニットテスト（`test/conflictDetection.test.ts`）で検証する。Gantt（`_doApplyGanttData`）・Flowchart（`_doWrite`）・テンプレート挿入（`_insertTemplate`）の全書き込み経路を単一プリミティブ `_writeDocument` に集約する。base は `EditorPanel._baseText`（改行 LF 正規化済み）として保持し、`null` の間（初回同期前）はコンフリクト扱いせず初回書き込みを許可する。バックアップ退避（R-C09-04）のタイムスタンプ生成・命名規則（`<ファイル名>.conflict-<mine|remote>-<timestamp><ext>`）は VS Code API 非依存の純粋関数として `src/backupNaming.ts`（`backupTimestamp` / `backupFileName`）に分離し、ユニットテスト（`test/backupNaming.test.ts`）で検証する。ファイル書き込み自体（`workspace.fs.writeFile`）は `_backupConflict` 側に残す。

###### ＜base スナップショット管理＞

- ■■■ R-C09-01 ビューアの表示状態が「どのドキュメント内容（テキスト）からパースされたか」を base スナップショットとして記録する。`_sendUpdate`（ready／外部変更同期）でビューアを更新するたびに、その時点のドキュメントテキスト（改行は LF に正規化）が base として保持されること。base は書き込み成功後にも書き込んだ内容で更新されること（テスト済）

###### ＜コンフリクト検知＞

- ■■■ R-C09-02 全文置換による書き込みの直前に、ライブのドキュメント内容およびディスク実体を base と比較してコンフリクトを検知する。base と一致する場合のみ従来どおり書き込み＋保存すること。base と不一致（他者が並行変更）の場合は黙って上書きしないこと。改行コードのみの差異（CRLF/LF）はコンフリクトとして扱わないこと。自分の書き込みエコー（書き込もうとしている内容と現状が一致）はコンフリクトとして扱わないこと。共有ドライブで `TextDocument` がディスクより古い場合に備え、`vscode.workspace.fs.readFile` によるディスク読み取りも判定に併用すること（テスト済）
- ■■■ R-C09-05 ビューア操作中（`_isOperating = true`）に届いた外部変更も、操作完了後の書き込み前検知で必ずコンフリクトとして扱う。操作中は外部変更イベントを無視する（R-G07-04／R-F06-04）が、その間 base は更新されないため、操作完了後の全文置換前に base 比較に乗り、外部変更が検知・分岐されること。既存の `_applyQueue` 直列化が壊れないこと（テスト済）

###### ＜コンフリクト解決＞

- ■■□ R-C09-03 コンフリクト検知時はモーダル警告（`showWarningMessage`）でユーザーに解決方法を選択させる。「最新を読み込む（自分の編集は破棄）」選択時は他者の変更を保持してディスクから再同期し、自分の書き込みを中止すること。「自分の変更で上書き（他者の変更は破棄）」選択時のみ従来どおり全文置換すること。ダイアログを閉じる（未選択）場合は安全側（最新を読み込む）として扱うこと
- ■■■ R-C09-04 コンフリクト解決で破棄される側の内容を、衝突ファイルとしてバックアップ退避する。上書き選択時は他者の版を、最新読み込み選択時は自分の版を、`<ファイル名>.conflict-<mine|remote>-<timestamp>.md` として同じディレクトリに保存すること。バックアップ失敗はコンフリクト解決をブロックしないこと（ベストエフォート）（テスト済）

### R-C10 アクティブエディタ追従 #US-C10 #実装済v2.7.0

> **理由：** 複数のMermaid文書を行き来するとき、単一のビューアパネルを手動で開き直さず、現在作業中の対応文書へ安全に追従させるため。

> **説明：** 拡張機能は `window.onDidChangeActiveTextEditor` を監視し、設定 `mermaid.followActiveEditor` が有効な場合に限り、既に開いている単一 `EditorPanel` の対象文書を切り替える。追従対象は Markdown／Mermaid言語として開かれ、内容から Gantt または Flowchart と判別できるテキスト文書だけとする。切替は既存の `_applyQueue` と、直接のテンプレート挿入・明示保存を含む文書操作キューが完了してから実行し、旧文書向けの未完了編集を新文書へ適用しない。ADR 0003 の単一パネル、ADR 0005 の遅延分離、ADR 0006 の楽観的同時実行制御を維持する。

###### ＜追従対象と設定＞

- ■■□ R-C10-01 `mermaid.followActiveEditor` を boolean 設定として提供し、既定値を `true` とすること
- ■■□ R-C10-02 設定が有効で単一 `EditorPanel` が開いているとき、アクティブなテキストエディタが対応する Gantt／Flowchart 文書へ変わると、同じパネルがその文書へ追従すること。新しいパネルを追加で開かないこと
- ■■□ R-C10-03 Markdown／Mermaid言語であっても Gantt／Flowchart と判別できない文書、その他の言語のソース文書、Settings・Webview等の非テキストUIへ移動した場合は追従せず、現在の対象文書を維持すること
- ■■□ R-C10-04 `mermaid.followActiveEditor` が `false` の場合はアクティブエディタ変更で対象文書を切り替えないこと。コマンドによる明示的な `createOrShow` は従来どおり利用できること

###### ＜安全な文書切替＞

- ■■□ R-C10-05 文書切替は、その時点で保留中の `_applyQueue` と、`_writeDocument` による全文置換・テンプレート挿入・明示保存を追跡する文書操作キューが完了した後に行うこと。切替待ちの間に旧Webviewから届く新規操作を受け付けず、旧文書向けの編集・更新を新文書へ適用しないこと。短時間に複数の切替要求が発生した場合は最新要求を採用すること
- ■■□ R-C10-06 切替時は文書変更リスナーを新文書へ再登録し、保留中のdebounce更新を破棄すること。旧文書の全操作完了後に `_isOperating`・各操作キュー・`_baseText`・Ganttキャッシュ等の文書スコープ状態を初期化し、ダイアグラム種別・Webview内容・パネルタイトルを新文書に合わせて更新すること。処理中の旧文書操作を `_isOperating = false` で隠さないこと。タイトルには対象ファイル名を含めること
- ■■□ R-C10-07 コマンドから既存パネルを別文書へ再利用する `createOrShow` も、アクティブエディタ追従と同じ安全な切替経路を使用すること

## ＜ガントチャート＞

### R-G01 ガントチャート表示 #US-G01 #フェーズ1

> **理由：** Mermaidのgantコードを開いたとき、日程の全体像を視覚的に把握しながら編集できるよう、インタラクティブなガントチャートを横並びで確認したいから。

###### ＜パネル起動＞

- ■■□ R-G01-01 Markdown内のgantブロックに対してガントパネルを開けること。コマンド `mermaid.openGantt` またはエディタタイトルバーボタンからパネルが開くこと
- ■■□ R-G01-02 ガントパネルはエディタの横（`ViewColumn.Beside`）に表示される。既に開いている場合は再利用すること

###### ＜描画＞

- ■■□ R-G01-03 タスクを横棒（タスクバー）として時系列に描画する。開始日・期間に応じて正しい位置・幅で表示されること
- ■■□ R-G01-04 セクション単位でタスクをグループ化して表示する。セクション名が行ラベルとして表示されること
- ■■□ R-G01-05 日付軸（タイムライン）を上部に表示する。年月・週単位の目盛りが表示されること。表示範囲はタスク全体を包含すること
- ■■□ R-G01-06 タスクの状態（`done` / `active` / `crit` / 未指定）を色で区別する。各状態に対して視覚的に異なるスタイルが適用されること
- ■■□ R-G01-07 VS Codeのカラーテーマ（ダーク/ライト）に追従する。VS Code標準のCSS変数を使用していること
- ■■□ R-G01-10 タスクバー・マイルストーンのラベルがバー幅（またはマイルストーンの図形サイズ）に収まらない場合、省略（…）表示にせず、バー／マイルストーンの右側外側に全文を表示すること。マウスオーバー時のツールチップ（全文表示）は表示位置によらず維持すること

###### ＜同期・保存＞

- ■■□ R-G01-08 Mermaidコードの変更から500ms以内にパネルが更新される。デバウンス処理を実装し、連続入力中は更新を遅延させること
- ■■□ R-G01-09 ガントパネル上の操作完了時に `.md` ファイルを自動保存する。タスク移動・リサイズ・名称変更・追加・削除のいずれの操作後も、VS Codeのダーティマーカーが残らないこと。セクション並び替え・Undo等の構造編集（structuralEdit）後も同様に保存されること

### R-G10 ビューのパン・ズーム（ガント） #US-G10 #フェーズ1

> **理由：** 長期スケジュールや大量タスクでも全体を把握しながら作業したいから。

- ■■□ R-G10-06 `Ctrl` キーなしのマウスホイールは上下スクロールとして動作する。ブラウザ標準のスクロール動作（縦方向）を妨げないこと
- ■■□ R-G10-03 ズームの最小倍率・最大倍率を設ける。最小: 7px/日（月単位が見渡せる）、最大: 60px/日（日単位が見やすい）とすること
- ■■□ R-G10-07 ホイールズームの縮小（ズームアウト）下限を、全タスクの全期間がビューに一望できる倍率まで緩和する。全期間がビュー幅（ラベル列を除く）に収まる px/日 が `MIN_PPD`（7px/日）より小さい場合、その倍率までズームアウトできること。ただし潰れて操作不能にならないよう絶対下限（0.5px/日）を設けること。全期間が標準下限内に収まる通常ケースでは従来どおり `MIN_PPD` を下限とすること
- ■■■ R-G10-08 Windowsのディスプレイ拡大率が125%等の非整数 `devicePixelRatio` 環境でも、ラベル列（sticky）とタイムライン列のタスクバーの境界に視覚的な重なり・ズレが発生しないこと。`devicePixelRatio` に応じた物理px丸め補正（CSS px値を物理pxに変換して丸め、CSS pxへ戻す）をラベル列・タスクバー双方の座標計算に一貫して適用すること。`devicePixelRatio === 1`（拡大率100%）の環境では従来どおり重なりが発生しないこと（リグレッションなし）

###### ＜ビュー位置リセット＞

- ■■□ R-G10-04 ツールバーの「全体表示」ボタンでガントチャート全体が画面に収まるよう自動調整する。全タスクが収まる倍率・スクロール位置に自動調整されること（推奨）
- ■■□ R-G10-05 ツールバーの「リセット」ボタンでビューの表示位置・拡大率を初期状態に戻せる。初期状態はデフォルト倍率（デフォルトpx/日）、スクロール位置は「今日 − 3日」が左端に見えること。パネル起動時・コード更新時も同じ位置に自動スクロールすること

### R-G19 タスク日程の終了日形式書き戻し #US-G19 #フェーズ7

> **理由：** ガント編集で日程を変更したタスクを Mermaid ソースへ書き戻すとき、日数（`Nd`）よりも終了日（`YYYY-MM-DD`）の方が意図が明確で、他ツール・レビューでも読みやすいから。ただしファイルを開いただけで既存タスクの表記を一括変換すると差分ノイズが大きいため、編集したタスクだけを変換したい。

> **説明：** 内部モデルは従来どおり `duration`（日数）ベースを維持し、書き戻し表記だけをタスク単位フラグ `GanttTask.useEndDate` で切り替える。終了日は開始日 + duration の exclusive（Mermaid 既定・パーサ `duration = diffDays(start, end)` と一致）で算出する。

- ■■□ R-G19-01 `GanttTask` に `useEndDate?: boolean` を追加する。true のとき当該タスクを `開始日, 終了日(YYYY-MM-DD)` 形式でシリアライズし、未設定/false のときは従来どおり `開始日, Nd` 形式でシリアライズすること。終了日は開始日 + duration 日（exclusive）とすること
- ■■□ R-G19-02 Web エディタで日程を変更する編集（ドラッグ移動・リサイズ完了、日程編集ポップアップ適用、タスク編集ポップアップ適用、タスク新規追加、タスク複製）を行ったタスクに `useEndDate` を立てる。マイルストーンの追加・日程編集（ドラッグ移動・日程編集ポップアップ／タスク編集ポップアップ適用）・複製も対象に含める。ラベルのみ編集・状態変更・crit トグル・依存関係の設定/解除では立てないこと
- ■■□ R-G19-03 ファイルを開いただけ（未編集）のタスクは表記を変換しない。`structuralEdit` で全タスクを再シリアライズしても、`useEndDate` 未設定のタスクは `Nd` 形式のまま維持されること
- ■■□ R-G19-04 パーサは終了日形式（`開始日, 終了日`）と日数形式（`Nd`/`Nw`/`Nh` 等）の双方を今後も受理する。終了日形式でパースしたタスクには `useEndDate = true` を付与し、再オープン後も終了日形式を維持する（往復安定）。日数形式のタスクには付与しないこと
- ■■□ R-G19-05 `afterId`（`after <id>`）を持つタスクは `useEndDate` の値にかかわらず常に `after <id>, Nd` 形式で出力する（相対開始に終了日を組み合わせると Mermaid 描画が不安定になるため）。マイルストーン（`status: 'milestone'`, `duration 0`）は `afterId` を持たない場合、`useEndDate` 未設定（未編集）なら `開始日, 0d`、`useEndDate = true`（追加・編集済み）なら `開始日, 開始日`（duration 0 の exclusive 終了日 = 開始日と同日）で出力すること
- ■■□ R-G19-06 `inclusiveEndDates` ディレクティブは出力・追加しない（既存タスク全体の終了日解釈に影響するため）。パーサの終了日 exclusive 解釈（`duration = diffDays(start, end)`）は変更しないこと

### R-F04 エッジの追加・削除・ラベル編集 #US-F04 #フェーズ2

- ■■□ R-F04-10 新規エッジの既定線種をツールバーで設定できる。ツールバーの「既定エッジ」セレクタで実線矢印/点線矢印/太線矢印/矢印なし実線/矢印なし点線を選べ、ポートドラッグで追加するエッジがその線種で生成されること。この設定はMermaidコード（`.md`）には保存できないため**ビューア表示中のみ保持**する一時設定とし、`vscode.getState/setState` でパネルの再表示（hide/show・種別切替）をまたいで維持するが、パネルを閉じて開き直した場合は既定（実線矢印）に戻ること

- ■■■ R-F04-03 エッジラベル部分またはエッジ線のダブルクリックでラベルをインライン編集できる。確定後にMermaidコードが `A -->|ラベル| B` 形式で更新されること。ラベル未設定のエッジ（`.edgeLabel` が潰れている場合）でも、エッジ線のダブルクリックでエッジ中央付近に編集フィールドが必ず表示されること（テスト済）
- ■■■ R-F04-04 エッジラベルを空文字で確定するとラベルが除去される。`A -->|ラベル| B` から `A --> B` に更新されること（テスト済）
- ■■□ R-F04-08 エッジラベル要素とエッジの対応付けはDOM順インデックスに依存せず確実に行う。Mermaid 11 はラベルの内側 `<span>` にも `edgeLabel` クラスを付与するため、`.edgeLabel` 数がエッジ数の2倍になりDOM順では崩れる。ラベルグループの `data-id`（`L_{始点}_{終点}_{連番}`）から対象エッジを解決し、ラベルのダブルクリック編集が正しいエッジを対象とし、インライン入力欄が当該ラベル位置に表示されること。ラベル未設定エッジはエッジ線のダブルクリックでエッジ中央に編集欄を表示すること

###### ＜エッジ削除＞

- ■■■ R-F04-05 エッジをクリック選択後に `Delete` キーまたは右クリック「エッジを削除」でエッジを削除できる。エッジをクリックすると選択状態（強調表示）になること。MermaidコードからエッジR-F04行が除去されること（テスト済）

###### ＜エッジ特定の正確性＞

- ■■■ R-F04-06 エッジのラベル編集・スタイル変更・削除は対象エッジの始点／終点ノードを正しく特定して反映する。Mermaid 11 系では `edgePath` の `LS-`/`LE-` クラスが固定値（`a1`/`b1`）となるため、これに依存せず `path` の `id`（`L_{始点}_{終点}_{連番}` 形式）から始点・終点ノードを解決すること。複数エッジが存在しても意図したエッジのみが更新されること（テスト済）
- ■■■ R-F04-07 エッジ操作は始点→終点の向きまで一致する行のみを対象とする。エッジ行の端点を実際にパースし、`from→to` の向きが一致するエッジのみを変更・削除すること。逆向き（`B-->A`）や両ノードを参照するだけの無関係な行を取り違えないこと。並行エッジは登場順のインデックスで一意に特定すること（テスト済）
- ■■□ R-F04-09 エッジの選択・編集の当たり判定を実線幅より広く取る。エッジ線に沿った透明な太いヒット領域（約14px相当、ズームに追従）を重ね、細い実線上でなくてもクリック選択・ダブルクリック編集・右クリックができること。ヒット領域上の操作で背景パンが誤発火しないこと

### R-F07 ビューのパン・ズーム（フローチャート） #US-F07 #フェーズ2

###### ＜ズーム操作＞

- ■■□ R-F07-02 マウスホイールで拡大・縮小できる。カーソル位置を中心にズームすること。`Ctrl` キーは不要

- ■■□ R-F07-06 拡大してもノード・エッジ・ラベルの文字がぼやけずにくっきり描画される。キャンバスを合成レイヤーへ昇格させてビットマップ拡大しないこと。ベクター SVG / foreignObject をズーム毎に再ラスタライズし、どの倍率でも文字が鮮明であること
