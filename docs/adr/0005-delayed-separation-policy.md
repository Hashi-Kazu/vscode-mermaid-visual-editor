# ADR 0005: MD記述の更新ポリシーを「遅延分離」に統一

**日付**: 2026-06-15（v2.1.0、commit be5b9ef）  
**ステータス**: 採用済み  
**確信度**: 高

---

## コンテキスト

Mermaid フローチャート記法は、ノードをエッジ行へインライン記述する形式（`A[開始] --> B{条件}`）と、ノードを単独行で宣言しエッジはID参照のみとする分離形式（`A[開始]` / `A --> B`）の双方を標準としてサポートする。

ノード編集（ラベル変更・形状変更）を実装した際、同一ノードがインライン記述で複数回出現すると、どの出現を更新すべきかが不明確になり、不整合が生じた。

## 決定

本拡張は **分離形式（ノード単独宣言＋ID参照エッジ）** を「正の更新ポリシー」として採用する（要求仕様書 R-FP-01〜03）。

具体的には：
- **エッジの追加** は常に ID 参照のみで行う（`A --> B` 形式、ノードの囲み記号をエッジ行へ書き込まない）
- **ノードのラベル編集・形状変更** は、対象ノードを単独宣言行へ集約してから反映する（「遅延」分離）
- **既存行・サブグラフ・コメント** は保持する（全面正規化は行わない）

## 理由

コミットメッセージ（be5b9ef）に明記: "複数インライン出現による不整合を解消"。

全面正規化（既存ファイルを全て分離形式に書き直す）は、編集していないノードまで変更してしまい、git diff が巨大になる。「遅延分離」は編集したノードだけをその場で分離形式に変換するため、diff を最小化しつつ不整合を解消できる。

## 捨てた選択肢

- **インライン形式を維持**: 同一ノードが複数箇所に出現した場合の更新箇所特定が困難。
- **全面正規化**: 読み込み時にファイル全体を分離形式に変換する。実装はシンプルだが、ファイル内容の大幅な書き換えが発生し、git 履歴が汚染される。

## 影響

- `flowchartSerializer.ts` に `getNodeDef` / `stripInlineNodeDefs` / `upsertNodeDecl` ヘルパーを追加。
- 既存ファイルがインライン形式で書かれていても正常に動作する（ただし編集すると対象ノードが分離形式に変換される）。
- `sample/flowchart.md` は分離形式で記述されており、このポリシーの見本として機能する。
