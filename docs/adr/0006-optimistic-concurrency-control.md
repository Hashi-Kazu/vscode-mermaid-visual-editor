# ADR 0006: 全文置換書き込みに楽観的同時実行制御（コンフリクト検知）を適用

**日付**: 2026-06-17（v2.3.0、commit 8b459cc）  
**ステータス**: 採用済み  
**確信度**: 高

---

## コンテキスト

Gantt・Flowchart ビューアはいずれも編集操作のたびにドキュメント全文を再シリアライズして `vscode.WorkspaceEdit` で置換する（per-operation マージは行わない）。共有ドライブや Git pull による並行変更がある場合、全文置換は並行編集を無言で上書きする危険がある。

## 決定

**楽観的同時実行制御** を採用する。`EditorPanel._baseText` に「現在表示中の内容の基点テキスト（LF 正規化済み）」を記録し、全文置換直前にライブ/ディスク内容と比較する。

- **一致**: 並行変更なし → 書き込み続行
- **不一致かつ書き込み予定内容でもない**: 並行変更あり → モーダルで「最新読み込み」または「上書き」を選択させる。破棄側は `.conflict-*` バックアップに退避。

全書き込み経路（Gantt 編集・Flowchart 編集・テンプレート挿入）を単一プリミティブ `_writeDocument` に集約することで、チェックの漏れを防ぐ。

## 理由

コミットメッセージ（8b459cc）に明記: "共有ドライブ／Git経由の並行編集に対する楽観的同時実行制御を実装"（US-C09）。

per-operation マージ（OT/CRDT 方式）は実装コストが大きく、Mermaid の行指向テキスト構造では衝突解決ルールが複雑になる。楽観的制御（書き込み直前の比較）は実装がシンプルで、衝突が稀な共有ドライブ環境には十分。

CRLF/LF 差異（Windows の autocrlf 設定や共有ドライブ経由の変換）が誤検知を起こさないよう、比較前に改行を LF に正規化する。

## 捨てた選択肢

- **per-operation マージ**: 実装コストが大きい。Mermaid 行指向構造での衝突解決ルールが複雑。
- **ロック方式**: VS Code の Workspace API にはファイルロックの仕組みがなく、実現困難。

## 影響

- `src/conflictDetection.ts` に純粋関数として判定ロジックを分離（ADR 0007 参照）。
- Gantt の並び替え直後の書き込みズレ解消のため、Gantt 書き込みは `_enqueueGantt` で直列化（v2.3.2、フェーズ4.2）。
